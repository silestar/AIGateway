# 插件系统接口规范（完整版）

## 1. 设计原则

- **安全隔离**：插件以独立进程（Sidecar）运行，通过 HTTP 与主系统通信。主系统通过 Token 认证调用。
    
- **语言无关，Go 优先**：官方提供 Go SDK 及示例，社区可使用任何语言实现，但推荐 Go 以保持技术栈统一。
    
- **热插拔**：支持在线启用/禁用插件，无需重启主系统。
    
- **轻量高效**：插件进程启动快、资源消耗低，与主系统松耦合。

## 2. 通信协议

### 2.1 传输层

- 协议：HTTP/1.1（未来可扩展至 gRPC）
    
- 地址：`http://127.0.0.1:<plugin_port>`
    
- 内容类型：`application/json; charset=utf-8`

### 2.2 认证

每个插件注册后由主系统生成 `PLUGIN_AUTH_TOKEN`（加密存储于数据库），启动时以环境变量注入。主系统调用插件的所有请求均需携带 Header：

```text
Authorization: Bearer <plugin_auth_token>
```

插件必须验证 Token，不匹配返回 `401 Unauthorized`。

### 2.3 超时与重试

- 主系统调用插件钩子的超时默认 **5 秒**（可通过配置调整）。
    
- 超时或网络错误视为插件调用失败，主系统执行兜底策略（跳过该钩子或拒绝请求，由具体钩子决定）。
    
- 主系统不重试对插件的调用。

## 3. 插件描述文件 (manifest.json)

插件根目录必须包含此文件，供主系统识别和加载。

```json
{
  "name": "content-filter",
  "version": "1.0.0",
  "description": "过滤请求中的敏感词",
  "author": "dev@example.com",
  "binary": "plugin",
  "port": 9001,
  "hooks": ["pre_request"],
  "config_schema": {
    "type": "object",
    "properties": {
      "blocked_words": { "type": "string", "description": "逗号分隔的敏感词列表" }
    }
  }
}
```

- `binary`：插件可执行文件名（Go 编译产物常见默认值为 `plugin`）。
    
- `port`：插件监听的本地端口。
    
- `hooks`：插件订阅的钩子列表。
    
- `config_schema`：定义管理员可在 WebUI 配置的参数（JSON Schema），安装插件后由主系统传递。
    

## 4. 钩子（Hook）定义与调用格式

### 4.1 可用钩子

|钩子名称|触发时机|是否可修改|用途|
|---|---|---|---|
|`pre_request`|认证完成、路由之前|可修改请求|内容过滤、参数校验、自定义认证|
|`post_response`|上游响应返回、发送给客户端前|可修改响应|内容过滤、格式转换、加解密|
|`on_log`|请求结束后，日志写入前|不可修改|自定义计费、外部审计日志|
|`account_select`|账号选择前（`SelectAccount` 执行之前）|可过滤候选列表|预过滤候选账号列表（从列表中移除不可用账号），不直接指定账号|

### 4.2 通用调用格式

主系统向插件发送 POST 请求：

```text
POST /hook/<hook_name>
Content-Type: application/json
Authorization: Bearer <token>

{
  "key_id": 123,
  "consumer_name": "app1",
  "model": "gpt-4",
  "request": {
    "headers": { "content-type": "application/json" },
    "body": { ... }
  },
  "response": {                           // 仅在 post_response / on_log 存在
    "status_code": 200,
    "headers": { "content-type": "application/json" },
    "body": { ... }
  },
  "channel_id": 5,                       // 部分钩子携带
  "account_id": 12,
  "candidate_accounts": [                 // 仅 account_select 携带
    {"id":12, "priority":1, "status":"active"},
    {"id":15, "priority":2, "status":"active"}
  ],
  "config": { "blocked_words": "word1,word2" }
}
```

### 4.3 钩子响应规范

#### `pre_request`

- **放行**：`{ "action": "continue" }`，可选携带 `modified_request` 字段局部覆盖请求。
    
- **修改**：`{ "action": "continue", "modified_request": { "headers": {...}, "body": {...} } }`
    
- **拒绝**：`{ "action": "reject", "status_code": 403, "message": "内容被过滤" }`

#### `post_response`

- 返回 `{ "action": "continue", "modified_response": { "headers": {...}, "body": {...}, "status_code": 200 } }`
    
- 或 `{ "action": "reject", ... }`

#### `on_log`

- 异步调用，主系统不等待响应，返回内容被忽略。

#### `account_select`

- **不干预**：`{ "action": "use_default" }`，保持候选列表不变，主系统按默认优先级+粘性策略选择。
    
- **预过滤**：`{ "action": "filter", "exclude_ids": [12, 15] }`，从候选账号列表中移除指定账号。主系统在过滤后的列表上执行默认的优先级+粘性选择，保持路由的确定性。

> **设计说明**：`account_select` 钩子在账号池 `SelectAccount` 之前执行，作为预过滤步骤。插件不直接指定最终使用的账号，仅从候选列表中移除不想要的账号。这样既保留了插件的自定义能力，又不破坏路由引擎的确定性原则。

## 5. Go 官方 SDK

提供 `pkg/plugin/sdk` 包，简化插件开发。

### 5.1 SDK 方法

```go
// 获取必需环境变量，若不存在则 panic
MustGetEnv(key string) string

// 注册钩子处理函数
HandleHook(hookName string, handler HookHandler)

// 启动 HTTP 服务，自动挂载 /health、/hook/*
StartPlugin(port string, token string)

// 钩子处理函数签名
type HookHandler func(req HookRequest) HookResponse
```

### 5.2 示例插件：敏感词过滤

```go
package main

import (
    "strings"
    "github.com/your-ai-gateway/plugin-sdk"
)

func main() {
    token := sdk.MustGetEnv("PLUGIN_AUTH_TOKEN")
    port := sdk.MustGetEnv("PLUGIN_PORT")

    var blockedWords []string
    if config := sdk.MustGetEnv("PLUGIN_CONFIG"); config != "" {
        // 简单解析配置，实际可使用 JSON unmarshal
        blockedWords = strings.Split(config, ",")
    }

    sdk.HandleHook("pre_request", func(req sdk.HookRequest) sdk.HookResponse {
        userText := req.Request.Body["messages"].(string) // 根据实际模型结构调整
        for _, word := range blockedWords {
            if strings.TrimSpace(word) != "" && strings.Contains(userText, word) {
                return sdk.Reject(403, "包含敏感词")
            }
        }
        return sdk.Continue()
    })

    sdk.StartPlugin(port, token)
}
```

## 6. 插件生命周期

### 6.1 注册与安装

- 管理员上传 ZIP 压缩包（包含 `manifest.json` 和二进制文件），调用 `POST /api/plugins/upload`。
- 临时文件使用 `os.CreateTemp` 生成随机路径，避免并发覆盖，处理完成后自动清理。
- 系统解压至 `plugins/<name>/`，校验 `manifest.json` 必填字段（name、version、hooks），读取并存入数据库。
- 插件状态为 `installed`。

### 6.2 启动

- 管理员点击"启动"，调用 `PUT /api/plugins/:id/status { action: "start" }`。
- 主系统执行 `binary` 文件，作为子进程运行。
- 注入环境变量：
    - `PLUGIN_AUTH_TOKEN`
    - `PLUGIN_PORT`
    - `PLUGIN_CONFIG` (JSON 字符串，由前端表单生成)
- **启动后健康确认**：`cmd.Start()` 后同步等待最多 3 秒，轮询 `/health` 端点（最多重试 3 次，间隔 1 秒）。成功标记 `running`，失败则终止进程并标记 `error`。
- **进程退出监听**：异步 `cmd.Wait()` 监听进程退出，自动更新状态为 `stopped` + 记录退出原因（exit code / signal）。
- 主系统开始周期性健康检查。

### 6.3 状态枚举

插件状态统一为：`installed` / `running` / `stopped` / `unhealthy` / `error`。

| 状态 | 含义 | 前端标签颜色 |
|------|------|-------------|
| `installed` | 已安装，未启动 | 灰色 |
| `running` | 子进程运行中，健康检查通过 | 绿色 |
| `stopped` | 已停止（手动停止或进程退出） | 黄色 |
| `unhealthy` | 子进程在但健康检查失败 | 橙色 |
| `error` | 安装/启动等不可恢复错误 | 红色 |

### 6.4 健康检查

主系统每 10 秒 `GET /health`，期望返回 `{"status":"ok"}`。连续 3 次失败则标记 `unhealthy` 并自动禁用，输出告警日志。

### 6.5 停止与卸载

- 停止：调用 `PUT /api/plugins/:id/status { action: "stop" }`。发送 `POST /admin/shutdown`（带鉴权），等待 3 秒后强制终止进程。
- 卸载：调用 `DELETE /api/plugins/:id`。先停止，再删除插件目录及数据库记录。

## 7. 安全与错误处理

- **网络隔离**：插件端口仅绑定 `127.0.0.1`。
    
- **资源限制**：推荐通过 Docker 容器运行插件，限制 CPU/内存。
    
- **钩子错误降级**：任何钩子调用失败（超时、5xx、401），主系统记录警告并跳过该插件。`pre_request` 或 `post_response` 拒绝错误会直接终止请求。
    
- **敏感信息保护**：插件不得明文记录 Token 或请求内容，日志由主系统统一控制脱敏。
    
- **加密存储**：插件 Token 在数据库中使用全局 `SECRET_KEY` 加密存储。

## 8. WebUI 管理功能

- **插件列表**：卡片网格布局，展示名称、版本、状态（installed/running/stopped/unhealthy/error）、描述、作者、钩子标签、端口/PID。
- **上传安装**：点击"上传插件"按钮，选择 .zip 文件，调用 `POST /api/plugins/upload`。
- **启动/停止**：`PUT /api/plugins/:id/status { action: "start"/"stop" }`，带确认弹窗。
- **配置编辑**：弹出 JSON 编辑器，根据 `config_schema` 动态生成，保存后调用 `PUT /api/plugins/:id/config`。
- **渠道级配置**：配置弹窗内的"渠道配置"Tab，为每个渠道单独设置配置（覆盖全局配置）。
  - 列出已有渠道级配置，可编辑/删除
  - 可添加新的渠道级配置（输入渠道 ID）
  - API：`GET /api/plugins/:id/channel-configs`、`PUT /api/plugins/:id/channel-configs/:channelId`、`DELETE /api/plugins/:id/channel-configs/:channelId`
- **卸载**：二次确认弹窗，调用 `DELETE /api/plugins/:id`。
- **市场浏览**：点击"浏览市场"按钮，从远程注册中心拉取插件列表并一键安装。
  - 前提：需在系统设置中配置 `plugin_registry_url`
  - API：`GET /api/plugins/registry/list`、`POST /api/plugins/registry/install`
  - 注册中心列表缓存 5 分钟，安装成功后自动清除缓存
- **渠道类型**：`GET /api/plugins/channel-types` 返回所有可用渠道类型（内置 + 插件注册的），前端渠道类型下拉框动态获取。
- **空状态**：无插件时显示引导提示。

## 9. 插件分发与构建

- **开发**：插件作者使用 Go（或其他语言）编写，提供 `manifest.json` 和可执行文件。
    
- **打包**：`zip plugin.zip plugin manifest.json`
    
- **分发**：可通过文件上传，或未来对接插件市场。

本规范基于核心系统完全 Go 化设计，确保了高性能、轻量级和统一的技术栈，同时保留了对其他语言的开放性。

## 10. 远程插件注册中心

### 10.1 配置项

| 配置键 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `plugin.plugin_registry_url` | string | `""` | 注册中心 API 地址（如 `https://registry.example.com/plugins.json`），为空时不启用 |
| `plugin.use_registry_auth` | bool | `false` | 注册中心是否需要认证（透传当前用户的 Authorization Header） |

> **迁移说明**：旧字段 `marketplace_url` / `marketplace_token` 已替换为上述新字段。

### 10.2 API 接口

#### `GET /api/plugins/registry/list`

从远程注册中心拉取插件列表，5 分钟内存缓存。

**响应**：

```json
{
  "data": [
    {
      "name": "gemini-adapter",
      "version": "1.0.0",
      "description": "Google Gemini API adapter plugin",
      "author": "community",
      "download_url": "https://registry.example.com/plugins/gemini-adapter/1.0.0/download",
      "homepage": "https://github.com/example/gemini-adapter",
      "tags": "[\"gemini\",\"adapter\"]",
      "min_agw_version": "1.0.0"
    }
  ]
}
```

#### `POST /api/plugins/registry/install`

从注册中心下载并安装插件。

**请求**：

```json
{
  "name": "gemini-adapter",
  "download_url": "https://registry.example.com/plugins/gemini-adapter/1.0.0/download"
}
```

**流程**：下载 ZIP → 保存临时文件 → 调用已有 Install 流程 → 清除缓存

**响应**：同 `POST /api/plugins/upload`

## 11. 渠道类型插件

### 11.1 概述

第三方插件可以注册新的渠道类型（如 Gemini），动态扩展 AGW 的适配器能力。插件启动后，AGW 自动探测并注册。

### 11.2 发现协议

插件启动并通过健康检查后，AGW 调用：

```text
GET /.well-known/channel-type
Authorization: Bearer <plugin_auth_token>
```

如果插件不提供渠道类型，返回 `404 Not Found`，AGW 静默跳过。

如果提供，返回：

```json
{
  "type": "gemini",
  "name": "Google Gemini",
  "base_url": "https://generativelanguage.googleapis.com",
  "description": "Google Gemini API adapter"
}
```

| 字段 | 必填 | 说明 |
|------|------|------|
| `type` | 是 | 渠道类型标识，如 `gemini`、`cohere` |
| `name` | 是 | 显示名称 |
| `base_url` | 否 | 默认上游地址，前端创建渠道时可自动填充 |
| `description` | 否 | 描述信息 |

### 11.3 注册流程

1. 插件启动 → 健康检查通过
2. AGW 调 `/.well-known/channel-type` 探测
3. 如返回 200，解析 `type` 字段
4. 调用 `adapterregistry.RegisterChannelType()` 注册元数据
5. 前端通过 `GET /api/plugins/channel-types` 获取完整渠道类型列表

### 11.4 渠道类型 API

#### `GET /api/plugins/channel-types`

返回所有渠道类型（内置 + 插件注册的）。

**响应**：

```json
{
  "data": [
    { "type": "openai", "name": "OpenAI", "is_plugin": false },
    { "type": "anthropic", "name": "Anthropic", "is_plugin": false },
    { "type": "gemini", "name": "Google Gemini", "is_plugin": true, "base_url": "https://generativelanguage.googleapis.com", "description": "Google Gemini API adapter" }
  ]
}
```

### 11.5 前端集成

- 渠道类型下拉框（Channels.vue）从 API 动态获取，不再硬编码
- 选择插件注册的渠道类型时，自动填充 `base_url`（如果插件提供了默认值）
- 插件注册的渠道类型显示 `(Plugin)` 后缀标识
