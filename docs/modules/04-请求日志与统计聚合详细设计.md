# 请求日志与统计聚合详细设计

## 1. 概述

本模块负责完整记录每一次 AI 调用的详情，并基于日志预聚合产生多维度统计数据，为运营分析、成本核算和故障排查提供数据支撑。设计目标：

- **高性能写入**：请求结束后的日志记录不得阻塞主流程，采用异步批量写入。
    
- **灵活查询**：支持按密钥、渠道、账号、模型、状态、时间等多维度组合筛选。
    
- **多维聚合**：预聚合系统、密钥、渠道三级日统计，加速前端图表加载。
    
- **可配置生命周期**：原始日志可按保留策略自动清理或归档。
    

## 2. 日志数据结构

### 2.1 原始请求日志表 (request_logs)

```sql
CREATE TABLE request_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    timestamp DATETIME NOT NULL,
    log_type VARCHAR(20) DEFAULT 'consumption',  -- 日志类型: consumption/probe/health_check
    trace_id VARCHAR(64),                         -- 请求追踪ID，贯穿整条故障转移链
    client_ip VARCHAR(45),                       -- 客户端IP地址
    key_id INT NOT NULL,
    model_name VARCHAR(100) NOT NULL,
    channel_id INT,
    account_id INT,
    retry_chain JSON,               -- 故障转移记录
    is_stream BOOLEAN DEFAULT FALSE,
    prompt_tokens INT DEFAULT 0,
    completion_tokens INT DEFAULT 0,
    status_code SMALLINT,
    error_msg TEXT,
    latency_ms INT,
    cost DOUBLE DEFAULT 0,          -- 费用估算（基于模型费率计算）
    request_meta JSON,              -- 请求摘要（脱敏）
    response_meta JSON,             -- 响应摘要（脱敏）
    has_detail INTEGER DEFAULT 0,   -- 是否有详细请求/响应内容文件（0/1）
    INDEX idx_consumer_time (key_id, timestamp),
    INDEX idx_channel_time (channel_id, timestamp),
    INDEX idx_model_time (model_name, timestamp),
    INDEX idx_timestamp (timestamp),
    INDEX idx_log_type (log_type, timestamp)
);
```

- `retry_chain` JSON 示例：

```json
[
  {"channel_id":3, "account_id":12, "started_at":"2026-05-07T10:00:00Z", "latency_ms":1200, "status_code":429, "error":"429"},
  {"channel_id":3, "account_id":15, "started_at":"2026-05-07T10:00:01Z", "latency_ms":30000, "status_code":504, "error":"timeout"},
  {"channel_id":7, "account_id":28, "started_at":"2026-05-07T10:00:31Z", "latency_ms":850, "status_code":200, "result":"success"}
]
```

每个 `RetryChainEntry` 包含以下字段：

- `channel_id` / `account_id`：尝试的渠道与账号
- `started_at`：本次尝试开始的 ISO8601 时间戳
- `latency_ms`：本次尝试耗时（毫秒）
- `status_code`：上游返回的 HTTP 状态码
- `error`：错误摘要（成功时为空，含 `result: "success"`）


- `request_meta` / `response_meta`：可配置脱敏的请求/响应摘要，默认关闭详细正文，仅记录 `model`, `messages count` 等元信息。

### 2.2 日志类型 (log_type)

`log_type` 字段用于区分不同来源的请求日志，确保统计聚合仅包含真实业务数据：

| log_type 值 | 说明 | 记录来源 |
|---|------|---------|
| `consumption` | 用户业务请求（默认值） | 代理请求处理完成后异步写入 |
| `probe` | 账号探测请求 | `account.Manager` 通过 `onProbeDone` 回调通知，写入 `stats.RequestLog(log_type=probe)` |
| `health_check` | 管理员手动测试渠道请求 | `ChannelHandler.TestChannel` 成功/失败均记录 |

**设计要点**：

- 默认值为 `consumption`，确保未显式设置类型的日志归入业务统计。
- 探测日志：`account.Manager` 在探测完成后通过 `onProbeDone` 回调通知外部，`main.go` 中注册回调写入 `stats.RequestLog`，`log_type` 设为 `probe`。
- 测试日志：`ChannelHandler` 注入 `asyncWriter`，`TestChannel` 成功/失败都记录 `health_check` 日志，便于管理员回溯测试历史。
- **聚合过滤**：所有统计聚合查询（系统/密钥/渠道三级）均添加 `WHERE log_type = 'consumption'` 条件，排除探测和测试请求对业务统计的干扰。
- **实时计数器过滤**：Redis 实时计数器同样过滤 `probe` 和 `health_check` 类型的日志，确保"今日实时"指标仅反映真实业务流量。

### 2.3 Token Usage 提取

系统通过 `pkg/usage/usage.go` 模块从上游响应中真实提取 token 用量，而非依赖估算：

- **支持格式**：OpenAI / Anthropic / Gemini 三种响应格式自动识别解析。
- **非流式请求**：从响应体 `usage` 字段直接提取 `prompt_tokens` 和 `completion_tokens`。
- **流式请求**：从 SSE 流的最后一条 chunk 中提取 token usage（OpenAI 格式为 `stream_options.include_usage=true` 触发的最终 usage chunk）。
- **测试请求**：`TestChannel` 同样提取 token usage，记入 `health_check` 日志。

### 2.4 费用计算

系统提供 `estimateCost()` 函数，基于模型名匹配费率表计算每次请求的估算费用：

- **费率匹配**：按模型名关键字匹配费率，支持 GPT-4/GPT-4o/GPT-3.5 系列、Claude-3/Claude-3.5 系列、Gemini 系列等主流模型。
- **计算公式**：`cost = prompt_tokens × 输入单价 + completion_tokens × 输出单价`，单价以每百万 token 计。
- **扩展性**：费率表当前硬编码于代码中，后续可接入配置文件或数据库配置，支持管理员自定义费率。
- **记录时机**：费用在日志写入时同步计算并存入 `cost` 字段，供后续统计聚合使用。

## 3. 异步写入机制

### 3.1 架构

```text
请求结束 (HTTP Handler)
    → 构造 RequestLog
    → 发送至缓冲 channel (容量可配，如 10000)
    → 日志写入协程批量消费 channel
    → 事务批量 INSERT 到 request_logs 表
    → 同时更新 Redis 实时计数器（原子 INCRBY）
```


- 使用 Go 的 `chan *RequestLog`，避免分配大量 goroutine。
    
- 批量大小：每 50 条或每 100 毫秒刷一次，取两者先满足的条件。
    
- 数据库连接池独立，避免影响主业务。

### 3.2 优雅关闭

系统收到 SIGTERM 时：

1. 停止接收新请求。
    
2. 关闭 channel，等待已入队的日志全部刷入数据库（超时 5 秒）。
    
3. 退出。

### 3.3 失败处理

- 写入数据库失败时，重试 3 次指数退避，若仍失败，将日志写入本地文件（`logs/failed_requests.log`）作为兜底，并触发告警。

## 4. 实时计数器 (Redis)

用于"今日"级指标实时展示：

- `stats:system:total_requests:2026-05-04` → INCRBY
    
- `stats:system:success_requests:2026-05-04`
    
- `stats:system:fail_requests:2026-05-04`
    
- `stats:consumer:<id>:total_requests:2026-05-04`
    
- `stats:channel:<id>:total_requests:2026-05-04`  
    每个键 TTL 设为 48 小时，由夜间的聚合任务在读后计算准确值更新统计表。

> **注意**：实时计数器仅对 `log_type = 'consumption'` 的日志进行递增，`probe` 和 `health_check` 类型的日志不会影响实时计数器。

前端需要"今日实时"数据时，API **仅查询 Redis 计数器**，不合并聚合表数据。聚合表仅用于历史查询（昨天及之前的数据）。每日凌晨的聚合任务将前一日的 Redis 数据最终固化为聚合表记录。

## 5. 聚合调度器

### 5.1 任务设计

一个 Go 协程定时执行（每 5 分钟一次），执行以下逻辑：

- 对当前日期的 `request_logs` 数据（**仅 `log_type = 'consumption'`**）按 `key_id, model_name` 和 `channel_id, model_name` 汇总：
    
    - 总请求数，成功数（status_code 2xx），失败数，平均延迟，总 tokens，活跃账号数（distinct account_id）。
        
- 写入 `*_daily_stats` 表（使用 UPSERT 语法，因为同一日可能多次执行）。

> **重要**：三个聚合查询（系统/密钥/渠道）均添加 `WHERE log_type = 'consumption'` 过滤条件，确保探测（`probe`）和测试（`health_check`）请求不计入业务统计。

### 5.2 执行频率与容错

- 每 5 分钟一次，确保日统计数据在日间也能逐步更新。
    
- 若任务执行中再次触发（如上一轮未执行完），通过分布式锁（Redis `SET NX`）保证单实例执行。
    
- 执行失败记录错误日志，下一周期自动重试（针对当前日期数据全量刷新，幂等）。
    

## 6. 查询 API 设计

### 6.1 实时指标 API

`GET /api/stats/realtime`  
返回示例：

```json
{
  "date": "2026-05-04",
  "total_requests": 15230,
  "success_requests": 14800,
  "fail_requests": 430,
  "active_consumers": 45,
  "active_channels": 12
}
```

数据来源：**以 Redis 计数器为准**（今日实时增量数据）。聚合表仅用于历史查询（昨天及之前的数据），不参与今日实时数据计算。

### 6.2 密钥统计 API

`GET /api/stats/consumer?key_id=5&start=2026-04-01&end=2026-05-04&granularity=daily`  
返回：按天的数组，每项包含 `date`, `total`, `success`, `fail`, `avg_latency`, `tokens`，可筛选模型。

后端逻辑：直接查询 `consumer_daily_stats` 表，按时间/模型过滤，利用索引快速返回。

### 6.3 渠道统计 API

`GET /api/stats/channel?channel_id=3&granularity=weekly`  
若粒度是周/月，对日统计表进行动态聚合：`SELECT SUM(total_requests) GROUP BY WEEK(date)`

### 6.4 请求日志查询 API

`GET /api/logs/requests?key_id=5&model=gpt-4&status=failed&start=...&end=...&page=1&page_size=20`  
返回分页结果，每条包含重试链简要信息，点击可获取完整细节（单独查询 `retry_chain` 字段）。

前端导出功能：同一查询条件加上 `export=csv` 参数，后端流式输出 CSV 下载。

### 6.5 请求详情内容文件

**`GET /api/logs/:id/detail`**

获取请求/响应完整内容（JSON 文件存储）。仅当数据库 `has_detail=1` 时可用。

存储方案：
- 代理转发成功后，异步捕获原始请求体（JSON）和响应（headers + status），序列化为 JSON 文件
- 文件路径：`logs/requests/{YYYY}/{MM}/{DD}/{trace_id}.json`
- 异步写入：goroutine 执行，先写文件再更新 DB 的 `has_detail=1`，不阻塞主请求路径
- 脱敏：请求 headers 中的 `Authorization` 仅存前缀+后缀掩码
- 配置开关：`detail_log_enabled`（默认 `true`），可通过热重载 API 关闭

## 7. 数据生命周期管理

### 7.1 原始日志保留

- 默认保留 **30 天**，可通过 `LOG_RETENTION_DAYS` 环境变量调整。
    
- 定时清理协程：每小时执行，扫描 `logs/requests/` 目录树和 `logs/YYYY/MM/` 目录树，删除超过保留天数的文件，并清理空目录。
    
- 对于 SQLite，执行后需 `VACUUM` 回收空间。

### 7.2 统计聚合表保留

- 系统日统计保留 **2 年**。
    
- 密钥/渠道日统计保留 **1 年**，之后自动删除。

### 7.3 归档（可选扩展）

- 对超期但需保留的原始日志，可异步导出为 Parquet/JSON 文件存储到对象存储或本地归档目录，导出后删除数据库记录。

## 8. 性能考量与优化

- **索引**：确保在经常查询的列上有复合索引，如上文所示。
    
- **分区**：在 MySQL/PostgreSQL 中可按月分区 `request_logs` 表，提高删除和查询效率。
    
- **异步不等待**：请求处理绝不阻塞于日志写入，确保主路径低延迟。
    
- **统计表小数据量**：日聚合后数据量极小，前端查询可快速响应。

## 9. 与全局加密的关系

- `request_logs` 中不存储任何敏感密钥，`retry_chain` 仅记录账号 ID，不包含明文 Key。
    
- 如需记录脱敏后的密钥 Key 后缀（如后四位），可在写入日志时从 Consumer 模块获取。

## 10. 系统运行日志增强

### 10.1 跨日自动滚动

`dailyWriter` 实现跨日自动滚动机制：

- 写入时检测当前日期是否与文件日期一致，若跨日则自动切换到新日期文件。
- 文件路径格式保持 `logs/{year}/{month}/{day}.log`，无缝切换无需重启。

### 10.2 文件大小限制

- 新增 `max_size_mb` 配置项，限制单个日志文件的最大体积。
- 当文件大小超过限制时，自动切换到新文件（文件名添加序号后缀，如 `day_2.log`）。
- 配置示例：`max_size_mb = 100`（默认 100MB）。

### 10.3 探测与测试日志记录流程

**探测日志**：

```text
account.Manager 探测完成
    → 触发 onProbeDone 回调
    → main.go 中注册的回调函数构造 RequestLog (log_type=probe)
    → 写入 asyncWriter 缓冲 channel
    → 批量刷入 request_logs 表
```

**测试日志**：

```text
管理员触发 TestChannel
    → ChannelHandler 执行测试请求
    → 成功/失败均构造 RequestLog (log_type=health_check)
    → 通过注入的 asyncWriter 写入缓冲 channel
    → 批量刷入 request_logs 表
```

这份设计处理了从生成、异步落地、聚合到清理的完整链路，确保了系统在处理高并发请求时，日志和统计不会成为瓶颈。
