## 分阶段实施策略（基于依赖与约束）

### 阶段划分总览

```text
阶段1：基础设施层（无外部API依赖）
  └─ 项目骨架、配置、数据库、加密、日志系统
       ↓
阶段2：代理引擎与渠道适配（少量外部API调用）
  └─ HTTP转发、流式处理、OpenAI适配器
       ↓
阶段3：核心业务层 - 账号池（逻辑密集，外部调用可控）
  └─ 优先级选择、粘性绑定、故障降级、按需探测
       ↓
阶段4：核心业务层 - 路由与密钥（纯逻辑，无外部依赖）
  └─ 分组管理、模型存在性过滤、密钥认证
       ↓
阶段5：管理API与WebUI（无外部API依赖）
  └─ RESTful API、 Vue前端、模型自动发现
       ↓
阶段6：统计日志与可观测性（批量处理，低频率调用）
  └─ 异步日志、实时计数、日聚合、请求追踪
       ↓
阶段7：扩展与完备（可并行，独立模块）
  └─ 插件系统、国际化、Docker部署、其他适配器
```

### 各阶段详细说明

---

#### 阶段 1：基础设施层

**目标**：搭建可独立运行的空骨架，零外部 API 依赖。

**约束风险**：无。完全本地执行。

**任务清单**：

- 项目目录与文件骨架创建
    
- 配置加载模块（Viper + .env）
    
- 存储抽象层（GORM + SQLite 默认）
    
- 全局加密服务（AES-256-GCM，SECRET_KEY 自动生成）
    
- 系统日志模块（zap，按日归档，本地文件写入）
    
- Gin 路由骨架 + 基础中间件（Recovery/CORS）
    
- DI 容器初始化（uber/dig）
    
- 空 Web 壳（Vue 项目骨架）

**AI 实施要点**：可一次性给出全部文件结构，AI 批量生成，无需中断等待上游。

**里程碑**：`go build` 成功，服务可启动，日志正常输出，配置文件加载正常。

---

#### 阶段 2：代理引擎与渠道适配

**目标**：实现最小请求转发链路，验证架构可行性。

**约束风险**：**需要上游 API Key**，存在限流风险。建议使用免费/测试 Key，且仅做单次验证。

**任务清单**：

- HTTP 代理引擎（连接池、超时、透明转发）
    
- 流式传输支持（SSE → 客户端透传，零拷贝）
    
- OpenAI 适配器实现（Chat Completions 接口）
    
- 最小硬编码转发链路：`Gin → Proxy → Adapter → Upstream`
    
- 请求/响应日志雏形（控制台输出即可）

**AI 实施要点**：

- 适配器代码可参考 OpenAI 官方文档生成，首次验证只需 1-2 次 API 调用。
    
- 若被限流，可切换到 mock 模式先验证逻辑。
    

**里程碑**：通过硬编码的账号，成功调用 OpenAI 并返回结果（含流式）。

---

#### 阶段 3：账号池核心逻辑

**目标**：实现最复杂的业务逻辑模块，但对外部 API 的调用可控。

**约束风险**：探测恢复功能需调用上游 API，但可**延时验证**——先实现纯逻辑单元测试，最后再实际调用一次验证。

**任务清单**：

- 账号模型与 CRUD API
    
- 优先级选择逻辑（priority ASC）
    
- 用户粘性绑定（Redis `consumer_account_affinity`）
    
- 故障降级（连续失败计数 → disabled）
    
- 按需探测恢复（含 min_disable_duration 冷却保护）
    
- 渠道级冷却与可用性事件发布
    
- 账号解密缓存（`account_key_cache`）
    
- 单元测试覆盖（mock upstream）

**AI 实施要点**：

- **核心逻辑密集但对外部API依赖低**，大部分可通过 mock 测试验证。
    
- 探测恢复的真实 API 调用可单独后置，先用单元测试保证状态机正确。
    
- 这是最复杂的模块，建议逐个子功能提交，避免 AI 上下文溢出。

**里程碑**：单元测试全部通过，账号状态机在各种故障场景下行为正确。

---

#### 阶段 4：路由与密钥模块

**目标**：实现分组管理和智能路由，将账号池与请求链路打通。

**约束风险**：无外部 API 依赖，纯逻辑实现。

**任务清单**：

- 密钥管理模块（CRUD、API Key 认证、分组权限）
    
- 渠道分组与密钥分组管理
    
- 模型存在性过滤（基于 `channel_models` 表）
    
- 分层确定性路由引擎（Group 权重 → 渠道权重 → 账号优先级）
    
- 密钥-分组-路由串联 API
    
- 分组管理单元测试

**AI 实施要点**：

- 本阶段无外部API调用，可密集生成代码。
    
- 路由逻辑的测试用例需覆盖：空账号渠道跳过、同权重渠道选择、分组耗尽等场景。

**里程碑**：完整请求链路打通——密钥认证→分组路由→账号选择→转发→返回。

---

#### 阶段 5：管理 API 与 WebUI

**目标**：构建完整的管理界面，让系统可被可视化操作。

**约束风险**：模型自动发现（FetchModels）需调用上游 API 一次验证，可后置。

**任务清单**：

- 完整 RESTful 管理 API（密钥/渠道/账号/分组/统计）
    
- Vue 前端页面开发（仪表盘、密钥、渠道/账号、分组、请求日志、系统设置）
    
- 模型自动发现功能（FetchModels 接口 + UI 交互）
    
- 前后端联调

**AI 实施要点**：

- 前端页面可逐页生成，先做静态 UI 再对接 API。
    
- FetchModels 的首次验证可能被限流，可先 mock 返回数据完成 UI，最后再真实调用一次。

**里程碑**：可通过 WebUI 完成完整的密钥创建、渠道配置、分组设置操作。

---

#### 阶段 6：统计日志与可观测性

**目标**：实现完整的日志记录与多维度统计分析。

**约束风险**：无外部 API 依赖。日志写入本地/数据库，统计聚合为后台任务。

**任务清单**：

- 请求日志异步批量写入（channel + batch insert）
    
- `retry_chain` JSON 完整记录
    
- Redis 实时计数器
    
- 日聚合定时任务（系统/密钥/渠道维度的统计表）
    
- 统计查询 API
    
- 请求日志查询 API（含筛选、分页、导出）
    
- 仪表盘实时数据对接
    
- 日志清理定时任务
    

**AI 实施要点**：

- 异步写入架构需注意 goroutine 泄漏和优雅关闭。
    
- 聚合任务可用分布式锁保证单实例执行。
    

**里程碑**：请求日志可追溯每次调用的完整重试链，仪表盘展示实时与历史数据。

---

#### 阶段 7：扩展与完备

**目标**：完成插件系统、国际化、部署方案和其他渠道适配器。

**约束风险**：其他适配器需各服务商 API Key 测试，可逐个验证，互不阻塞。

**任务清单**（可并行）：

- 插件系统（Sidecar 管理、钩子调用）
    
- Go 插件 SDK
    
- 国际化（中英文语言文件）
    
- Docker 多阶段构建 + docker-compose
    
- Claude、Gemini 等适配器
    
- 用户教程站点（guide/ VitePress）
    
- README 最终完善
    

**AI 实施要点**：

- 各适配器可独立并发生成，互不依赖。
    
- 插件系统的钩子协议已在设计文档中明确定义，可直接按规范生成。
    

**里程碑**：`docker compose up -d` 一键启动，全功能可用。

---

## 汇总

|阶段|核心内容|上游 API 依赖|主要风险|
|---|---|---|---|
|1|基础设施|无|低|
|2|代理引擎|⚠️ 少量验证|限流|
|3|账号池|⚠️ 可选验证|逻辑复杂度|
|4|路由与密钥|无|低|
|5|管理 API + WebUI|⚠️ FetchModels|联调|
|6|统计日志|无|数据量|
|7|扩展完备|⚠️ 各适配器|分散|

**核心原则**：

- 阶段 1-4 构建核心链路，阶段 5-7 为锦上添花。
    
- 每个阶段结束后都有可运行、可验证的产物。
    
- 需要上游 API 调用的任务集中在阶段 2 和适配器开发中，且均可被 mock 替代先行验证。
    
- AI 辅助时，每个阶段可以作为一个独立上下文，任务明确，依赖清晰。
